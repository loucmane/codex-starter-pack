# Task ID: 129

**Title:** Add Aegis Doctor, Repair, and Idempotency Hardening

**Status:** done

**Dependencies:** 127 ✓, 128 ✓

**Priority:** medium

**Description:** Implement a read-only diagnostic and explicit safe repair foundation for Aegis state so installs, starts, logs, verification, closeout, and interrupted workflows can be replayed or recovered safely. Document the target workflow model, add CLI/MCP doctor and repair surfaces where appropriate, and preserve Taskmaster and Serena as optional integrations.

**Details:**

Codebase analysis summary: Aegis is a Python package declared in `pyproject.toml` with console scripts `aegis` and `aegis-mcp-server`. The deterministic core lives in `scripts/_aegis_installer.py`; package CLI handlers live in `aegis_foundation/cli.py`; the repository wrapper mirrors Aegis commands in `scripts/codex-task`; MCP tools and resources are registered in `aegis_mcp/server.py`; public and packaged docs are mirrored under `docs/aegis/` and `aegis_foundation/assets/docs/aegis/`; regression coverage is concentrated in `tests/meta_workflow_guard/test_aegis_installer.py`, `test_aegis_mcp_server.py`, `test_aegis_installer_fixtures.py`, `test_aegis_cross_project_smoke.py`, and `test_aegis_invocation_contract.py`. Existing core surfaces include `inspect_project`, `status`, `next_action`, `plan_install`, `install`, `initialize_project`, `start_local_work`, `kickoff`, `log_work`, `verify`, `repair_handoff`, `closeout`, and `format_closeout_summary`. Task 127 added deterministic handoff repair through `repair_handoff` and MCP `aegis.handoff_repair`; Task 128 added concise closeout output, `--json`, completed current-work state, and post-closeout readiness acceptance. Build this task on those surfaces rather than creating a parallel implementation. First, document the target state model in a new source doc such as `docs/aegis/state-recovery-model.md` and mirror it into `aegis_foundation/assets/docs/aegis/`; update `docs/aegis/invocation-contract.md`, `docs/aegis/public-adoption-flow.md`, `docs/aegis/update-rollback.md`, and matching packaged copies. The model should define states such as `not_installed`, `invalid_manifest`, `installed_no_current_work`, `in_progress_ready`, `workflow_scaffold_incomplete`, `pending_tracking`, `strict_verification_required`, `closeout_required`, `completed_closeout`, and `stale_or_broken_pointers`; it should define which commands are read-only, which commands write evidence reports, which repairs require explicit apply confirmation, and how replay behaves for `install`, `init`, `start`, `kickoff`, `log`, `verify`, `closeout_ready`, and `closeout`. Second, add a read-only core diagnostic function in `scripts/_aegis_installer.py`, for example `doctor(target_dir, source_root, include_repair_plan=True)`, that reuses existing helpers such as `_read_json`, `_validate_with_schema`, `_managed_assets`, `_strict_managed_files_check`, `_strict_current_work_checks`, `_strict_pending_tracking_check`, `_strict_integration_checks`, `_closeout_passed`, and `next_action`. The doctor report should be structured and deterministic: `schema_version`, `target_root`, `read_only`, `checked_at`, `status`, `summary`, `current_state`, `checks`, `repair_plan`, and `next_action`. It should check manifest presence/schema/version, managed-file presence, local CLI shim existence/executable bit, workflow template presence, current-work JSON shape and status, `sessions/current` and `plans/current` symlink health, referenced session/plan/work-tracking/report paths, work-tracking surfaces, pending tracking queue state, latest verification and closeout reports, and optional integration state. It must treat absent `.taskmaster` and `.serena` as passing unless `current_work.integrations.<name>.required` is true. Third, add a safe repair core function, for example `repair(target_dir, source_root, apply=False, scopes=None)`, that consumes the doctor plan and applies only deterministic low-risk actions. Safe actions should include recreating missing `sessions/current` and `plans/current` symlinks from `.aegis/state/current-work.json`, recreating expected empty directories such as the active reports directory, restoring missing Aegis-managed runtime files from packaged/source assets only when the path is absent and the manifest identifies the file as managed, fixing executable bits for managed scripts, and normalizing completed closeout metadata only when both current work and `.aegis/reports/closeout-report.json` already prove a passed closeout. Repair must not overwrite divergent user files, must not delete or archive stale active folders by default, must not clear non-empty pending tracking, and must return actionable guidance for unsafe or manual-review cases. Mutating repair should write a repair report such as `.aegis/reports/repair-report.json`; dry-run repair should be read-only and include `would_apply` actions. Fourth, expose command surfaces. Add package CLI handlers and parser entries in `aegis_foundation/cli.py` for `aegis doctor --target-dir .` and `aegis repair --target-dir . --apply`, with concise default output and `--json` for full structured reports where useful. Mirror the same subcommands in `scripts/codex-task` under its `aegis` command. Add MCP tools in `aegis_mcp/server.py`, update `V1_TOOL_NAMES`, and follow existing response conventions: `aegis.doctor` is read-only, while `aegis.repair` previews by default and requires `apply=true` for mutation. Update prompts/resources only if they materially help agents choose doctor/repair after `aegis.next` or failed closeout. Fifth, harden replay semantics in the existing core functions. `install` and `initialize_project` should remain safe to repeat, preserving user files and avoiding backup sidecars. `start_local_work` and `kickoff` should detect existing in-progress current work: replay of the same task/title should return an `already_started` or equivalent no-op report with existing paths, a different in-progress task should be refused with a structured next action, and completed closeout state should allow starting new work. Partial workflow state should be recoverable by doctor/repair without silently allocating a second local task id or active folder. `log_work` should avoid duplicate S:W:H:E entries and duplicate plan evidence when a log command is replayed after its pending event was already consumed and the matching entry is already present. `verify` should remain safe to repeat and should not introduce duplicate manifest report entries or pending tracking ambiguity. `closeout` and `closeout_ready` should treat completed closeout state as terminal and replay-safe: repeated dry-run and final closeout should pass from completed state, avoid false readiness failures, and avoid unnecessary current-work or closeout-report churn unless an explicit refresh mode is added. Keep all implementation in the Aegis core and existing adapters; do not introduce a mandatory Taskmaster or Serena dependency.
