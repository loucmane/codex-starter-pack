# Task ID: 119

**Title:** Prove Local-First Aegis Package and MCP Installation Before PyPI

**Status:** done

**Dependencies:** 118 ✓

**Priority:** medium

**Description:** Build and validate local wheel and sdist artifacts for `aegis-foundation`, then prove a fresh project can register and run the Aegis MCP workflow from only the local artifact or package-style local project path. This task makes local artifact installation the required gate before any TestPyPI or PyPI publishing work.

**Details:**

Codebase analysis summary: `pyproject.toml` already defines the Python 3.11 distribution `aegis-foundation` with console scripts `aegis = aegis_foundation.cli:main` and `aegis-mcp-server = aegis_mcp.server:main`; `MANIFEST.in` and `[tool.setuptools.package-data]` include `aegis_foundation/assets/**`. The deterministic installer/runtime source of truth is `scripts/_aegis_installer.py`, and package invocation delegates through `aegis_foundation/cli.py`. MCP startup and tools live in `aegis_mcp/server.py` via `AegisMCPConfig`, `V1_TOOL_NAMES`, `RESOURCE_URIS`, and `PROMPT_NAMES`. Native MCP registration support lives in `aegis_foundation/mcp_registration.py` with package, pinned, GitHub, wheel, and source modes, surfaced through `aegis mcp generate-registration`, `execute-registration`, and `verify-registration`, with mirrored repo-local argparse support in `scripts/codex-task`. Existing tests already cover cross-project installs (`tests/meta_workflow_guard/test_aegis_cross_project_smoke.py`), invocation contracts (`test_aegis_invocation_contract.py`), release distribution metadata and env-gated local wheel smokes (`test_aegis_release_distribution.py`), native MCP registration (`test_aegis_native_mcp_registration.py`), and MCP target flows (`test_aegis_mcp_e2e_targets.py`). The remaining gap is a mandatory, fresh-folder, local-artifact proof that exercises registration plus the full workflow through closeout without depending on `/home/loucmane/codex`, `AEGIS_SOURCE_ROOT`, repository-local `scripts/codex-task`, `.taskmaster/`, or `.serena/`.

Implementation approach:
1. Create normal Task 119 work-tracking artifacts and reports under the current repository workflow before changing code. Record the local-first release decision: no TestPyPI/PyPI publishing task may start until the local artifact proof report passes.
2. Harden local artifact production. Use the existing `aegis certify-release` path in `scripts/_aegis_installer.py` as the preferred release-candidate authority. Ensure it builds both wheel and sdist using `uv build --sdist --wheel --out-dir <dist>` when `uv` is available, falls back to `python -m build` only when appropriate, computes SHA-256 checksums, inspects wheel/sdist contents for required package assets, entry points, schemas, MCP server package, workflow templates, Claude gate scripts, and packaged docs, and writes durable evidence to a report such as `reports/aegis-local-install-proof/local-artifact-proof.json` or the existing release certification report path if that is the better local pattern.
3. Make local artifact and local package-path MCP registration first-class. In `aegis_foundation/mcp_registration.py`, verify that `--source-mode wheel --artifact <wheel>` and the local project/path mode generate `uvx --from <artifact-or-path> aegis-mcp-server --default-target-dir . --transport stdio` without requiring `--source-root` or a repository checkout. Add path validation and structured errors for missing wheel, missing local project `pyproject.toml`, unsupported archive type, or artifact paths that resolve outside the caller-visible filesystem. Preserve argv-list execution with `subprocess.run(..., shell=False)` and keep manual config-file writes fallback-only.
4. Add a dedicated fresh-folder proof harness, likely `tests/meta_workflow_guard/test_aegis_local_artifact_mcp_proof.py`. The harness should build a local wheel/sdist into a temporary dist directory, create a brand-new temporary git project, clear `AEGIS_SOURCE_ROOT` and source-checkout `PYTHONPATH` leakage, register Aegis MCP using the native registration command generated from the local wheel or package path, start the MCP server through the generated `mcp_server_argv` with `mcp.client.stdio`, and prove the server reports packaged assets via `aegis-mcp-server --describe-config` or equivalent structured config evidence.
5. Exercise the complete workflow from the fresh target through MCP and/or the installed artifact command only: call `aegis.inspect`, `aegis.status`, `aegis.plan_install`, `aegis.install` with explicit apply, `aegis.kickoff`, `aegis.log` for scope evidence, make a small task-scoped source or docs change in the new git project, record S:W:H:E tracking with `aegis.log`, run `aegis.verify` including strict verification when closeout requires it, log verification evidence, then call `aegis.closeout` with the required acknowledgement/update-handoff semantics. Assert that `.aegis/foundation-manifest.json`, `.aegis/state/current-work.json`, `.aegis/reports/install-plan.json`, `.aegis/reports/install-report.json`, `.aegis/reports/verification-report.json`, and `.aegis/reports/closeout-report.json` exist and show passing status.
6. Prove no checkout dependence. After the fresh-folder proof, recursively scan target `.aegis/` state, reports, generated contract files, current session, current plan, and work-tracking surfaces to ensure they do not contain `/home/loucmane/codex` or any source checkout path unless the specific test is intentionally exercising local project-path mode and the path appears only in the native client registration command evidence, not in installed target runtime state.
7. Keep fake-client and live-client layers separate. Use deterministic fake `claude` and `codex` binaries in tests to prove native registration execution and verification without requiring real clients on CI. Add an optional live smoke, gated by explicit environment variables, that runs the same local wheel registration against real installed native clients when available; the local artifact proof itself should still be runnable on a developer machine without publishing anything.
8. Update docs and packaged doc mirrors in `docs/aegis/distribution.md`, `docs/aegis/mcp-client-setup.md`, `docs/aegis/release-policy.md`, `docs/aegis/release-verification-matrix.md`, and `aegis_foundation/assets/docs/aegis/*`. Document exact pre-publication commands for local wheel/sdist build, local wheel MCP registration, local project-path MCP registration, fresh-folder install/kickoff/log/verify/closeout, expected report locations, and the rule that TestPyPI/PyPI publishing is blocked until this proof passes. Keep public package snippets present but clearly downstream of this local proof.
9. Update CI/release templates only as documentation unless the repository already has a concrete CI workflow for Aegis release gates. If adding a command or report flag, prefer extending `aegis certify-release` rather than creating a separate release engine.
10. Preserve existing compatibility: do not break `scripts/codex-task aegis ...`, existing package CLI commands, existing MCP tool names, generated `.aegis/` paths, existing env-gated wheel smoke tests, or Task 111/115/117 target fixtures. The task is complete only when a brand-new temporary project can use the built local artifact plus the generated native registration command to complete inspect, install, kickoff, S:W:H:E log, verify, and closeout without cloning or invoking this repository checkout.
