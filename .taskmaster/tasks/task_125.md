# Task ID: 125

**Title:** Build Public Aegis Real-Project Adoption Flow

**Status:** done

**Dependencies:** 124 ✓

**Priority:** medium

**Description:** Create a polished public adoption flow that turns the existing low-level Aegis install, MCP registration, and kickoff mechanics into `task-master init`-style project setup plus natural-language project work. The canonical setup command must be `aegis init` run from inside the target project. `aegis setup` may exist only as an optional compatibility alias, not as the documented happy path. After `aegis init`, a fresh Claude session in that target project should be able to receive a normal request such as `Improve BrandMark accessibility` and rely on installed Aegis files/hooks to guide readiness, local task identity, S:W:H:E tracking, verification, closeout, and handoff without a large workflow checklist prompt.

**Details:**

Codebase analysis summary: this repository is the Python package `aegis-foundation` from `pyproject.toml`, with console scripts `aegis = aegis_foundation.cli:main` and `aegis-mcp-server = aegis_mcp.server:main`. The current CLI in `aegis_foundation/cli.py` exposes low-level commands such as `inspect`, `plan-install`, `install`, `status`, `next`, `kickoff`, `log`, `verify`, `closeout`, `certify-release`, and `mcp generate-registration/execute-registration/verify-registration`; it does not yet expose the public `init`, `start`, or positional `mcp register claude` flow. Shared deterministic behavior lives in `scripts/_aegis_installer.py` and must stay mirrored into packaged assets under `aegis_foundation/assets/scripts/_aegis_installer.py`; native MCP command rendering lives in `aegis_foundation/mcp_registration.py`, where `client_argv()` already renders `claude mcp add ... -- uvx --from <spec> aegis-mcp-server --default-target-dir . --transport stdio` and `run_native_client()` uses an argv list with no shell; MCP tools/resources/prompts live in `aegis_mcp/server.py`, where existing mutating tools require `apply=true`; current public docs live in `docs/aegis/distribution.md`, `docs/aegis/mcp-client-setup.md`, `docs/aegis/invocation-contract.md`, and `docs/aegis/live-acceptance-matrix.md` and currently lead with the longer inspect/plan-install/install/kickoff sequence. Existing installer behavior already preserves `CLAUDE.md` by merging an Aegis-managed block via `_merge_claude_entrypoint()`, creates `.aegis/state/current-work.json`, `sessions/current`, `plans/current`, active work-tracking files, hooks, reports, and a project-local `./.aegis/bin/aegis` shim. Existing `kickoff()` already marks `integrations.taskmaster.required=false` and `integrations.serena.required=false`, but it still requires explicit numeric `--task`, explicit `--slug`, and a git worktree. The public flow must layer over these proven internals instead of replacing them.

Implement the feature as an additive public UX layer over the existing core:
1. Define the public adoption contract and command names. Add a short design artifact, for example `docs/aegis/public-adoption-flow.md` plus the packaged mirror under `aegis_foundation/assets/docs/aegis/`. The canonical happy path must be documented as package/install once, optional native MCP registration, then per-project initialization and natural work: `aegis mcp register claude --scope user` (or simply `aegis mcp register claude` with Claude user scope as the default), `cd <target-project>`, `aegis init`, and later `aegis start "Improve BrandMark accessibility"` when a local tracked task is needed. The advanced/manual equivalents remain `aegis inspect`, `aegis plan-install`, `aegis install --apply`, `aegis mcp generate-registration`, `aegis mcp execute-registration`, `aegis mcp verify-registration`, and explicit `aegis kickoff --task ... --slug ... --title ...`. `aegis setup` may be implemented as an alias to `aegis init` only if useful, but it must not be the canonical public command in docs, prompts, generated instructions, or acceptance evidence.
2. Add a high-level `aegis init` command in `aegis_foundation/cli.py`, backed by shared helpers in `scripts/_aegis_installer.py`. The command must run from inside any target project with `--target-dir .` as the default and should mirror `task-master init` ergonomics: conservative defaults of `profile=generic`, `primary_agent=claude`, and `agents=[claude]` unless explicitly overridden. It should run the current inspect -> plan_install -> install/apply -> verify sequence, return structured JSON, write normal reports under `.aegis/reports/`, and stop safely on conflicts/manual-review operations using the same no-partial-write behavior as `install()`. Keep a dry-run/plan-only mode available for review, but acceptance requires plain `aegis init` to fully install the runtime into a clean target project without requiring users to know `plan-install`, `install`, `--primary-agent`, `--agent`, or `--apply`. If `aegis setup` is added, route it through the same handler and mark it as an alias in help output.
3. Add a polished native MCP registration wrapper `aegis mcp register <client>`, with `aegis mcp register claude` as the required public Claude path. It should delegate to `aegis_foundation/mcp_registration.py`, support package/pinned/GitHub/wheel/source modes, default to package assets for public use, default Claude scope to `user`, and support explicit `--scope project` where appropriate. It should generate the same structured payload as `registration_payload()`, execute the exact argv from `client_argv()` when registering, and verify registration through the existing native inspection logic when possible. Preserve the exact argv/no-shell behavior of `execute_registration()`. It must not manually edit `.mcp.json` as the happy path; fallback snippets remain documentation-only for clients without native registration. Preserve `aegis mcp generate-registration`, `aegis mcp execute-registration`, and `aegis mcp verify-registration` as advanced/debug surfaces.
4. Add local-task start mode for projects without Taskmaster. Expose the public command as `aegis start "normal task title"`, where the quoted title is a positional argument. Add a deterministic local task allocator, for example `.aegis/state/local-tasks.json`, that generates monotonically increasing numeric local task ids, slugs from titles, and collision-safe branch names. `aegis start "Improve BrandMark accessibility"` should allocate id/slug, then reuse the existing `kickoff()` machinery to create branch/state/session/plan/work-tracking files. Also support optional advanced `aegis kickoff --local --title "Improve BrandMark accessibility"` for users who want to stay on the kickoff surface, while preserving existing explicit `aegis kickoff --task <id> --slug <slug> --title <title>` behavior. The resulting `.aegis/state/current-work.json` must keep `integrations.taskmaster.required=false` and `integrations.serena.required=false` and must work when `.taskmaster/` and `.serena/` are absent.
5. Update installed project instructions generated by `_render_claude_entrypoint()` and `_render_contract()` so normal user language is enough. The installed `CLAUDE.md` should tell a fresh Claude session to run readiness/status/next, use `aegis init` only if the project is not installed, start local work with `aegis start "<inferred user request title>"` when no current work exists, infer a short title/slug from the user's ordinary request when reasonable, log scope before edits, use native tools for implementation, consume pending tracking with `aegis log --pending-id current --plan-step auto`, run task verification, run strict Aegis verification, close out, and only then report completion. Keep this guidance concise; it should replace the need for a giant per-session checklist prompt, not become one. Update generated `.claude/settings.json` allowlists so `aegis init`, `aegis start`, existing low-level workflow commands, and project-local `./.aegis/bin/aegis ...` equivalents are available where appropriate.
6. Preserve existing project instructions. Continue using the managed `<!-- AEGIS:BEGIN claude-runtime -->` block around Aegis instructions and preserve pre-existing `CLAUDE.md` content under `## Existing Project Instructions`. Do not introduce `.bak`, `.orig`, `.backup`, or similar sidecar files. Existing non-Aegis files must be preserved byte-for-byte except for the explicit, safe `CLAUDE.md` managed-block merge.
7. Strengthen readiness and mutation enforcement for the public flow. Ensure installed `.claude/scripts/pretooluse-gate.sh`, `posttooluse-tracking.sh`, `tracking-stop-gate.sh`, and `gate_lib.py` enforce: no persistent source mutation before READY/current work; only init/start/log/verify/closeout repair commands plus low-level install/kickoff repair equivalents are allowed while blocked; a task-scoped mutation creates pending S:W:H:E tracking; a second mutation and session stop are blocked until the pending event is logged; protected paths such as `CLAUDE.md`, `AGENTS.md`, `.aegis/`, `.claude/settings.json`, hook scripts, and generated workflow files are not manually mutated by normal implementation work.
8. Update MCP prompts/tool descriptions in `aegis_mcp/server.py` only where they help the public flow. Prefer a small number of clear prompts/descriptions that steer fresh sessions toward `aegis.status`, `aegis.next`, and public `aegis init` / `aegis start` concepts, while preserving the rule that prompts are advisory and reports/tool outputs are evidence. If MCP equivalents such as `aegis.init` or `aegis.start` are added, keep mutating-tool behavior consistent with existing MCP safety by requiring explicit `apply=true`; do not add MCP source-editing tools. Aegis remains the workflow control plane and native agent tools remain the implementation path.
9. Update docs and packaged mirrors. `docs/aegis/distribution.md`, `docs/aegis/mcp-client-setup.md`, `docs/aegis/invocation-contract.md`, `docs/aegis/live-acceptance-matrix.md`, and packaged copies under `aegis_foundation/assets/docs/aegis/` should show the public commands first: `aegis mcp register claude`, `aegis init`, and `aegis start "Improve BrandMark accessibility"`. Demote the long inspect/plan-install/install/kickoff/log sequence to advanced details. Include exact commands for fresh projects, existing projects with existing `CLAUDE.md`, local wheel/pre-publication use, package mode, native Claude/Codex MCP registration, local-task start without Taskmaster, optional `aegis kickoff --local --title`, and project-local shim fallback.
10. Maintain package/source parity. Any change to installer core, rendered instructions, docs, hooks, templates, command wrappers, or public command help must be reflected in packaged assets and validated by existing package-data/release tests. Do not break existing `scripts/codex-task aegis ...`, `aegis install`, `aegis kickoff`, `aegis log`, existing MCP tool names, or current release-candidate artifact workflows.
11. Treat acceptance as behavioral. The feature is not done because docs mention an easier flow; it is done only when a fresh and an existing real target can be initialized with `aegis init`, native Claude MCP registration works through `aegis mcp register claude`, local tracked work starts with `aegis start "Improve BrandMark accessibility"`, and a fresh Claude session can be given a normal short request such as `Improve BrandMark accessibility` while the installed Aegis runtime mechanically guides readiness, task identity, S:W:H:E tracking, verification, closeout, and handoff without a large workflow prompt.
