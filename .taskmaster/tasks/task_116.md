# Task ID: 116

**Title:** Aegis Strict Verification and Release Certification Pipeline

**Status:** done

**Dependencies:** 115 ✓

**Priority:** medium

**Description:** Add a strict Aegis verification gate and release-candidate certification workflow that mechanically proves installed target runtime completeness, packaging provenance, artifact installability, and portable operation without requiring Taskmaster or Serena.

**Details:**

Codebase analysis summary: Aegis currently centralizes installer/runtime behavior in `scripts/_aegis_installer.py`, exposes package CLI commands in `aegis_foundation/cli.py`, wraps the same core through MCP tools in `aegis_mcp/server.py`, packages runtime assets under `aegis_foundation/assets/`, and validates release/runtime behavior with `tests/meta_workflow_guard/test_aegis_release_distribution.py`, `tests/meta_workflow_guard/test_aegis_mcp_e2e_targets.py`, `tests/meta_workflow_guard/test_aegis_invocation_contract.py`, and `tests/meta_workflow_guard/test_aegis_mcp_server.py`. Current `aegis verify` validates the installed manifest schema plus manifest-declared gates and writes `.aegis/reports/verification-report.json`, but it is not yet a comprehensive strict runtime audit. Task 115 added the portable installed-project workflow layer: `aegis kickoff`, `.aegis/state/current-work.json`, `.aegis/bin/aegis`, packaged workflow templates under `.aegis/templates/workflow/`, PostToolUse/Stop mutation tracking, `aegis log`, and optional Taskmaster/Serena readiness semantics.

Implement the strict verifier in the shared Aegis core first, not as a CLI-only or MCP-only path. Add a strict verification entrypoint such as `verify(..., strict: bool = False)` or a dedicated `strict_verify(...)` in `scripts/_aegis_installer.py`, then thread it through `aegis_foundation/cli.py` as `aegis verify --strict --target-dir ...` and through `scripts/codex-task aegis verify --strict`. If the MCP surface is updated, extend `aegis_mcp/server.py` so `aegis.verify` accepts a `strict` boolean while preserving `acknowledge_report_write=true`, or add an explicitly named strict MCP tool only if that fits the existing `V1_TOOL_NAMES`/schema tests cleanly. Keep all surfaces delegating to the same core verifier.

Strict verification must emit a machine-readable report, preferably extending `.aegis/reports/verification-report.json` with `mode: "strict"`, a clear `status`, `summary`, and categorized `checks`, while preserving backward compatibility for the non-strict report shape. At minimum validate: manifest presence/schema/version alignment; `managed_files` existence and file-vs-directory correctness; executable bits for required scripts and `.aegis/bin/aegis`; installed local CLI shim usability by running or dry-checking `./.aegis/bin/aegis --version` without relying on repository-local `AEGIS_SOURCE_ROOT`; packaged workflow template completeness for every `AEGIS_WORKFLOW_TEMPLATE_NAMES` file under `.aegis/templates/workflow/`; hook wiring in `.claude/settings.json` for PreToolUse, PostToolUse, and Stop when Claude is enabled; protected-path behavior for both direct file writes and Bash redirections against adapter entrypoints such as `CODEX.md`; mutation tracking surfaces including `.aegis/state/current-work.json`, pending-tracking behavior, `aegis log` surface coverage, session/tracker/implementation/changelog/handoff/plan evidence paths, and refusal of mismatched pending evidence; report paths for install, verify, kickoff, and current-work state; read-only verification commands that redirect to `/dev/null` do not create pending tracking; and optional integration state where `.taskmaster/` and `.serena/` are reported as detected/absent but do not block strict verification unless the Aegis current-work payload explicitly marks them required.

Make strict checks mechanical. Avoid tests that only assert documentation text. For hook/protected-path/mutation behavior, reuse the installed-target execution helpers already present in `tests/meta_workflow_guard/test_aegis_mcp_e2e_targets.py` or extract small shared helpers if needed. The strict verifier should fail loudly with actionable check IDs such as `runtime.local_cli_shim`, `runtime.workflow_templates`, `claude.hook_posttooluse`, `claude.protected_path_write`, `mutation.pending_tracking`, `workflow.plan_evidence`, and `integrations.taskmaster_optional`. Required strict checks should make the CLI exit nonzero and MCP return the existing structured failed tool response style.

Add release-candidate certification as a first-class command or script. Prefer a package CLI command such as `aegis certify-release --out-dir reports/aegis-release-certification` or a repo wrapper under `scripts/codex-task aegis certify-release` if repository-only orchestration is more appropriate; either way, keep reusable logic in a small Python module/function instead of embedding a long shell script in CI. The certification workflow should build both wheel and sdist with the existing packaging contract from `pyproject.toml`, compute SHA-256 checksums for every artifact, capture provenance metadata including git commit, dirty-worktree status, package version, Python version, build command, artifact sizes, and timestamp, inspect wheel/sdist contents for required Aegis assets and entry points, then install from the built artifact into clean temporary target projects without `AEGIS_SOURCE_ROOT`. Use the existing fixture matrix from `tests/fixtures/aegis-target-projects/` and the target-shape helpers in `test_aegis_mcp_e2e_targets.py` as the model.

Certification smoke tests must run from installed artifacts, not from the source checkout: `aegis --version`, `inspect`, `plan-install`, `install --apply`, `status`, `verify --strict`, `kickoff`, readiness, a task-scoped mutation plus pending-tracking/logging cycle, and protected-path refusal. Include MCP certification where practical by launching `aegis-mcp-server` from the artifact with stdio, listing tools/resources/prompts, calling `aegis.inspect`, and calling strict verify through MCP if exposed. The output should be a deterministic JSON report under `reports/aegis-release-certification/`, for example `certification-report.json`, plus supporting text logs for build/install/test commands. The JSON report should include `schema_version`, `status`, `started_at`, `completed_at`, `artifacts`, `checksums`, `provenance`, `target_smokes`, `mcp_smoke`, `strict_verification_reports`, `failures`, and a publishing handoff section stating that GitHub release artifacts are the next publication step before PyPI.

Update documentation in `docs/aegis/distribution.md`, `docs/aegis/release-policy.md`, `docs/aegis/release-verification-matrix.md`, `docs/aegis/ci-install-templates.md`, and `docs/aegis/mcp-client-setup.md` only where the executable contract changes. Document `aegis verify --strict`, the certification report path, checksum/provenance expectations, clean-target smoke requirements, and the release handoff order: prepare GitHub release artifacts with attached wheel/sdist/checksums/provenance/certification report before any PyPI publication task. Keep portability explicit: Aegis can consume Taskmaster or Serena evidence if present, but strict verification and certification must pass in target projects that contain neither.

Add CI coverage. Extend `.github/workflows/ci.yml` or add a focused Aegis certification workflow so regular PR CI runs the strict verifier unit/integration tests and at least one non-env-gated artifact certification smoke. Keep expensive full matrix certification behind an explicit environment variable, workflow_dispatch, or targeted job if needed, but ensure regressions in `aegis verify --strict`, artifact checksums/provenance report generation, and clean install-from-artifact smoke cannot silently pass. Update existing workflow tests in `tests/meta_workflow_guard/test_ci_workflows.py` if this repository asserts workflow contents.

Acceptance: `aegis verify --strict` works from both installed package CLI and project-local `.aegis/bin/aegis`; strict reports fail on missing hooks, missing workflow templates, broken local shim, protected-path bypass, broken mutation tracking, missing full workflow surfaces, or required optional integration failure; certification builds wheel and sdist, computes checksums/provenance, installs from artifacts into clean targets, runs strict verification and runtime smokes, and writes a machine-readable report under `reports/aegis-release-certification/`; docs and CI reflect the new gate; Taskmaster and Serena remain optional in strict target verification; final handoff includes publishing guidance for GitHub release artifacts before PyPI.
