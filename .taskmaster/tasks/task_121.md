**Details:**

Codebase context from analysis: `pyproject.toml` packages `aegis-foundation` with console scripts `aegis = aegis_foundation.cli:main` and `aegis-mcp-server = aegis_mcp.server:main`; core workflow behavior lives in `scripts/_aegis_installer.py` and the packaged mirror `aegis_foundation/assets/scripts/_aegis_installer.py`; CLI wiring for `log`, `verify`, and `closeout` is in `aegis_foundation/cli.py`; MCP tool schemas and handlers are in `aegis_mcp/server.py`; installed hook behavior and pending tracking live in `.claude/scripts/gate_lib.py` and `aegis_foundation/assets/.claude/scripts/gate_lib.py`; current regression coverage is concentrated in `tests/meta_workflow_guard/test_aegis_installer.py`, `test_aegis_mcp_server.py`, `test_aegis_mcp_e2e_targets.py`, `test_aegis_release_distribution.py`, `test_aegis_native_mcp_registration.py`, and `test_aegis_invocation_contract.py`. Task 120 evidence in `docs/ai/work-tracking/archive/20260522-task120-fresh-local-install-proof-COMPLETED/reports/fresh-local-install-proof/claude-live-test-result.md` proved the intended install/kickoff/edit/log/verify/closeout path but exposed three UX gaps: agents had to know exact default surfaces, strict verify produced pending tracking that required exact handler/evidence copying, and closeout missing-evidence failures reported surfaces rather than actionable repair calls. Implementation approach: 1. Start Task 121 work-tracking normally and record Task 120 findings as scope evidence; keep PyPI/TestPyPI publication explicitly out of scope. 2. Replace the static `AEGIS_DEFAULT_LOG_SURFACES = ("implementation", "changelog", "handoff")` behavior in `scripts/_aegis_installer.py` with event-class-aware default surface selection while keeping explicit `surfaces` overrides backward compatible. Add a small helper such as `_infer_log_event_class(plan_step, handler, evidence, explicit_event_class)` and `_default_log_surfaces_for_event(event_class, evidence)` so `log_work` can default correctly when `surfaces is None`. Scope logs should update the always-written session and tracker plus `FINDINGS.md`, `DECISIONS.md`, `HANDOFF.md`, and the plan when `plan_step` is supplied. Implementation logs should update session, tracker, `IMPLEMENTATION.md`, `CHANGELOG.md`, `HANDOFF.md`, and the plan when supplied. Verification logs should update session, tracker, `IMPLEMENTATION.md` or `FINDINGS.md` as appropriate, `CHANGELOG.md`, `HANDOFF.md`, and the plan when supplied. Prefer explicit `event_class` support in CLI and MCP (`scope`, `implementation`, `verification`) and infer from `plan-step-scope`, `plan-step-implement`, `plan-step-verify`, handlers such as `aegis:verify`, and evidence such as `.aegis/reports/verification-report.json` only when omitted. 3. Extend `aegis_foundation/cli.py` and `aegis_mcp/server.py` so the CLI/MCP `aegis.log` surfaces expose the new defaults without breaking existing calls. Existing calls with repeated `--surface` or `surfaces: [...]` must keep exact behavior; omitted surfaces should use the new event-aware defaults. Update MCP schema tests that currently assert `surfaces.default is None` so they also assert the new `event_class` and pending-event input shape. 4. Improve pending-event ergonomics. Add support to `log_work` for consuming a pending event by id, for example CLI `aegis log --pending-id <id> --note ... --plan-step ... --plan-status ...` and MCP `aegis.log` input `pending_event_id`. If exactly one pending event exists, allow a documented `current` or `latest` sentinel only if the implementation can do so deterministically. When a pending id is provided, load `.aegis/state/pending-tracking.json`, copy that event's `handler` and `evidence` into the S:W:H:E entry, validate the event belongs to current work, clear only the consumed event, and reject ambiguous or unknown ids with a response that lists valid ids. Keep normal native implementation tools untouched; this is a logging/control-plane ergonomics change, not an MCP editor. 5. Update pending-event guidance in `_format_pending_tracking_for_error` and installed hook messages in `gate_lib.py` so blocking output includes a copyable Aegis repair shape such as `aegis log --pending-id <id> --note "<past-tense note>" --plan-step plan-step-implement --plan-status completed` plus the explicit handler/evidence fallback. 6. Improve closeout repair guidance in `closeout()` without weakening gates. Add a deterministic `repair_guidance` or `repair_plan` section to `.aegis/reports/closeout-report.json` whenever `closeout.evidence.*`, `closeout.plan.*`, `closeout.tracker.*`, or pending tracking gates fail. For missing evidence references, build a matrix from `evidence_matrix` and generate exact suggested `aegis log` calls when an existing S:W:H:E token for that evidence can be found on another surface and the handler/note can be inferred. If exact repair is not possible, emit a structured repair item with missing `surface`, `evidence`, suggested `event_class`, suggested `plan_step`, and placeholder handler/note fields rather than only reporting `details.missing`. Do not make closeout auto-log repairs; `--update-handoff` should remain limited to semantic handoff sections. 7. Keep source and packaged runtime in sync: mirror changes into `aegis_foundation/assets/scripts/_aegis_installer.py` and packaged hook/docs assets where applicable, and update installed docs/templates only where behavior changed, especially `docs/aegis/invocation-contract.md`, `aegis_foundation/assets/docs/aegis/invocation-contract.md`, generated Aegis instructions embedded in `_aegis_installer.py`, and any MCP docs/tests that describe `aegis.log`. 8. Add or adjust focused tests that reproduce the Task 120 live Claude path: install/kickoff into a representative web target, native edit creates pending tracking, one intended `aegis.log` call using pending-event consumption updates all required surfaces, strict verify passes, the strict verification log defaults update all closeout-required surfaces, and closeout passes without extra changelog-only relogging. 9. Record final Task 121 findings and verification evidence in active work-tracking, including whether these fixes remove the Task 120 publication blockers. Do not publish to TestPyPI or PyPI in this task.
