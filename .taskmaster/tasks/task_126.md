# Task ID: 126

**Title:** Harden Aegis Acceptance Fixture Verification

**Status:** done

**Dependencies:** 125 ✓

**Priority:** medium

**Description:** Replace brittle Aegis acceptance fixture assertions with runtime, AST, and structured semantic checks where possible, and document the few cases where a literal implementation style is intentionally required. Add regressions proving semantically correct fixture changes pass without forcing literal substring-oriented rewrites.

**Details:**

Codebase analysis summary: the repository is the Python package `aegis-foundation` from `pyproject.toml`, with console scripts `aegis = aegis_foundation.cli:main` and `aegis-mcp-server = aegis_mcp.server:main`. Public Aegis command wiring now lives in `aegis_foundation/cli.py` for `init`, `start`, `mcp register`, and the lower-level install/kickoff/log/verify/closeout flow; deterministic workflow behavior lives in `scripts/_aegis_installer.py` and must stay mirrored to `aegis_foundation/assets/scripts/_aegis_installer.py`; MCP tools and resources live in `aegis_mcp/server.py`; cross-project and target fixtures live under `tests/fixtures/aegis-target-projects/` and are exercised primarily by `tests/meta_workflow_guard/test_aegis_mcp_e2e_targets.py`, `test_aegis_installer.py`, `test_aegis_installer_fixtures.py`, `test_aegis_cross_project_smoke.py`, and `test_aegis_release_distribution.py`. The current acceptance-style web fixture test `test_installed_web_target_real_feature_change_updates_full_workflow` in `tests/meta_workflow_guard/test_aegis_mcp_e2e_targets.py` verifies the feature by literal source evidence such as the text `Add to cart` and a logged `rg` command, while also asserting exact S:W:H:E substrings across workflow surfaces. Task 125 live acceptance found that this kind of source-grep expectation can make semantically correct code look wrong and can pressure agents to rewrite code into a less idiomatic shape only to satisfy fixture literals.

Implementation approach:
1. Define an acceptance-fixture verification policy. Add a small design note, likely under `tests/fixtures/aegis-target-projects/README.md` or `docs/aegis/live-acceptance-matrix.md`, explaining that application/source behavior should be verified through runtime, AST, import, JSON-schema, or parsed semantic checks whenever practical. Literal substring checks are allowed only for Aegis-owned protocol contracts such as managed block markers, command names in docs/help, required report paths, schema keys, and S:W:H:E token syntax. If an application fixture genuinely requires a literal implementation style, document that style in the fixture contract and explain why a semantic check is not available.
2. Introduce reusable semantic assertion helpers instead of scattering source-grep assertions through individual tests. A good location is a new `tests/meta_workflow_guard/aegis_acceptance_assertions.py` module. Include helpers to parse Aegis workflow evidence into structured records, for example `parse_swhe_entries(text)`, `parse_plan_table(text)`, `assert_workflow_evidence(current_work, evidence=..., handler=..., task_id=..., slug=..., plan_step=...)`, and `assert_closeout_evidence_matrix(...)`. These helpers should validate the meaning of session, tracker, implementation, changelog, handoff, plan, strict verification, and closeout evidence without requiring exact whole-line substrings except where the Aegis protocol defines those tokens.
3. Add source-level semantic validators for the fixture languages already present. For Python/backend fixtures, use Python `ast.parse`, targeted import/runtime checks, or direct function execution where safe; for example verify that `app/main.py` still exposes a callable health surface returning the expected shape rather than searching for one exact return literal. For web fixtures, prefer an AST or runtime DOM-style check over `rg`. If the existing dependency stack does not provide a TypeScript/TSX parser, choose the smallest explicit test-only parser/runtime approach and document it; if no parser is introduced, limit literal expectations to the fixture contract and add a negative test proving comments or dead strings do not satisfy acceptance. Keep any parser dependency deliberate and isolated to tests.
4. Refactor `tests/meta_workflow_guard/test_aegis_mcp_e2e_targets.py` so application behavior checks are semantic. In `test_installed_web_target_real_feature_change_updates_full_workflow`, replace the direct `assert ... in source_text` and verification evidence that depends on `rg` with a helper that proves the resulting page behavior contains an actionable cart button or equivalent DOM state. The test should still verify that Aegis recorded the changed source path, that pending tracking was cleared, that `aegis verify --strict` passed, and that closeout evidence references implementation and verification evidence, but those checks should use parsed workflow records instead of brittle exact text lines wherever possible.
5. Add explicit regression fixtures for semantically correct but non-literal implementations. For the web target, include at least two equivalent source variants such as setting button content through a label variable, `append`, a helper function, or `appendChild` versus `append`, and prove both pass the semantic validator without requiring the exact `cartButton.textContent = ...` style. Add a negative variant where the expected text appears only in a comment or unused string and prove it fails. For the BrandMark-style accessibility case from Task 125, add a focused fixture or helper test that accepts semantically equivalent accessible markup, such as direct ARIA props or a documented prop-spread pattern, and rejects markup that only contains the expected words in dead code or comments.
6. Preserve meaningful literal checks for Aegis-owned contracts, but centralize them. Assertions for `<!-- AEGIS:BEGIN claude-runtime -->`, required CLI names such as `aegis init` and `aegis start`, required report paths like `.aegis/reports/verification-report.json`, and required workflow keys are still valid because they are public/runtime contracts. Make the tests distinguish those protocol literals from application implementation literals so future maintainers know which assertions are intentional.
7. Update docs and packaged mirrors only where the public acceptance contract changes. If `docs/aegis/live-acceptance-matrix.md` or other Aegis docs describe acceptance evidence, update the packaged mirror under `aegis_foundation/assets/docs/aegis/`. Do not rewrite unrelated distribution or invocation docs unless the test contract now requires a different public command or evidence standard.
8. Keep package/source parity intact. If any installer-rendered guidance, packaged docs, fixture expectations, or runtime assets change, mirror the change to the packaged asset tree and ensure release distribution tests still prove the wheel/package does not depend on `/home/loucmane/codex` at runtime.
9. Treat this as a test-quality and release-confidence task, not an Aegis UX redesign. Do not change `aegis init`, `aegis start`, MCP apply semantics, local task allocation, closeout behavior, or hook enforcement unless a fixture hardening test exposes a real behavioral bug; if a bug is found, isolate the fix in `scripts/_aegis_installer.py`, `aegis_foundation/cli.py`, `aegis_mcp/server.py`, or the relevant hook asset and add the smallest focused regression.
