# Task ID: 122

**Title:** Advance Aegis Workflow Guidance and Adapter Portability

**Status:** pending

**Dependencies:** 62 ✓, 121 ✓

**Priority:** medium

**Description:** Add a next-level Aegis workflow guidance layer that tells agents exactly what to do next, makes plan-step logging deterministic, and documents release/adaptor portability without changing the core architecture. Aegis MCP/CLI remains the workflow control plane, native agent tools remain the implementation path, and installed hooks/gates remain the enforcement layer.

**Details:**

Codebase analysis: `pyproject.toml` defines the `aegis-foundation` package with console scripts `aegis = aegis_foundation.cli:main` and `aegis-mcp-server = aegis_mcp.server:main`. Core deterministic behavior lives in `scripts/_aegis_installer.py` and must be mirrored into `aegis_foundation/assets/scripts/_aegis_installer.py`; CLI wiring lives in `aegis_foundation/cli.py`; MCP tools/resources/prompts live in `aegis_mcp/server.py`; installed Claude guidance is rendered by `_render_claude_entrypoint()` in the installer; packaged docs live under `docs/aegis/` and `aegis_foundation/assets/docs/aegis/`; live target fixtures already exist under `tests/fixtures/aegis-target-projects/{web-new,web-started,python-new,python-started,backend-new,backend-started}`. Task 121 already adds event-aware log defaults, `event_class`, `pending_event_id`, and closeout repair guidance; this task must explicitly build on that immediate-fix layer and keep longer-term release/adaptor work separated in a roadmap document. Implementation approach: 1. Add a read-only workflow guidance surface. Prefer an additive `aegis next` CLI command plus MCP tool `aegis.next`, and also embed the same `workflow_guidance` payload in existing `aegis.status` for backward-compatible discovery. Implement the core in `scripts/_aegis_installer.py`, for example `next_action(target_dir, *, source_root)`, so CLI and MCP share one deterministic path. The payload should include `phase`, `state`, `next_required_action`, `suggested_cli`, `suggested_mcp_call`, `missing_gates`, `copyable_repairs`, `read_only: true`, and `architecture_notes` that state Aegis controls workflow state while native agent tools perform implementation edits. It should cover at least these states: not installed -> inspect/plan-install/install; installed but no current work -> kickoff; current work without scope evidence -> log scope before editing; pending tracking exists -> log pending event; implementation evidence missing -> use native tools then log implementation; verification evidence missing -> run project verification then log verification; strict verification missing -> run `aegis verify --strict` then log its report; closeout gates incomplete -> run closeout-ready/dry-run repairs; closeout ready -> run strict closeout; closeout passed -> no workflow action required. 2. Add deterministic `plan_step=auto` support without introducing AI inference. Extend CLI `aegis log --plan-step auto`, MCP `aegis.log` input `plan_step: 'auto'`, and the shared `log_work()` path. Infer `plan-step-scope` only from explicit event class `scope`, a scope handler, or current guidance state that requires scope; infer `plan-step-implement` from a consumed pending mutation event or implementation event class; infer `plan-step-verify` from verification event class, verification handlers, app-level verification evidence, or `.aegis/reports/verification-report.json`. Treat app-level verification and strict verification as subtypes under the existing `plan-step-verify` row: app verification may append evidence without necessarily completing the step, while strict verification report evidence may complete it. If more than one inference is possible, fail with an actionable error listing explicit valid `--plan-step` values instead of guessing from free-text notes. Return `plan.inferred`, `plan.inference_reason`, and `plan.strict_verification_evidence` metadata in the log payload. Preserve explicit plan-step behavior exactly. 3. Improve MCP tool descriptions in `aegis_mcp/server.py`. Update docstrings/descriptions for `aegis.status`, `aegis.next`, `aegis.kickoff`, `aegis.log`, `aegis.verify`, and `aegis.closeout` so fresh Claude sessions naturally call status/next first, record scope before editing, use native tools for implementation, log pending mutation evidence, log verification before closeout, and never write `.aegis/` directly. Keep mutating MCP operations limited to Aegis workflow state and reports; do not add MCP source-editing tools. 4. Update installed Claude guidance generated by `_render_claude_entrypoint()` so the first visible operating loop is: run readiness/status/next, kickoff if needed, log scope with `--plan-step auto` before persistent edits, perform implementation with native Claude tools, consume pending tracking with `aegis log --pending-id current --plan-step auto`, run task verification, run strict verification, log strict verification with `--plan-step auto`, run closeout-ready/dry-run if blocked, then run strict closeout before claiming completion. Ensure `.claude/settings.json` allowlist guidance still permits `aegis status`, `aegis next`, `aegis log`, `aegis verify`, `aegis closeout`, and `./.aegis/bin/aegis ...` fallback calls. If adding a pre-mutation hook check for missing scope, implement it in `.claude/scripts/gate_lib.py` and packaged copies without deadlocking the `aegis log` repair command; otherwise document the scope-before-edit behavior as guidance and rely on closeout gates for enforcement. 5. Add a pre-closeout readiness surface. Either implement `aegis closeout --dry-run` or a separate `aegis closeout-ready` CLI/MCP read-only check; prefer sharing the same internal evaluator used by closeout so failure IDs and repair guidance match. The check should report missing plan steps, missing tracker checkboxes, pending tracking events, missing evidence references, missing strict verification evidence, and handoff semantic gaps before strict closeout. It must produce copyable repairs using Task 121 repair guidance, including `aegis log --pending-id <id> --plan-step auto`, explicit `aegis log --handler ... --evidence ... --surface ...` repairs when exact inference is possible, and structured placeholders when exact repair cannot be generated. It must not auto-log repairs or directly edit `.aegis/`. 6. Add MCP prompts for the requested workflow phases while preserving existing prompt names as aliases where useful. Add/update `PROMPT_NAMES` for `aegis.bootstrap`, `aegis.start_task`, `aegis.implement_task`, and `aegis.closeout_task`. `bootstrap` should drive inspect/status/plan-install/install/verify; `start_task` should drive status/next/kickoff/scope log; `implement_task` should state that native agent tools do implementation and Aegis logs workflow evidence; `closeout_task` should drive app verification, strict verification, verification logging, closeout-ready repairs, and closeout. Prompt text must say prompts are advisory and tool/report outputs are evidence. 7. Create a release-grade live acceptance matrix document, likely `docs/aegis/live-acceptance-matrix.md`, and package it under `aegis_foundation/assets/docs/aegis/`. The matrix must cover fresh and existing web, Python, and backend projects; explicit no-Taskmaster/no-Serena operation; a project that already has `.mcp.json`; native MCP registration; and CLI-shim fallback through `./.aegis/bin/aegis` when global `aegis` is unavailable. Each row should list starting project shape, command surface, expected next-action sequence, required reports, evidence paths, closeout criteria, accepted deferrals, and whether the row is automated, manual-live, or future release evidence. Update `docs/aegis/release-verification-matrix.md` to reference this matrix instead of duplicating it. 8. Add an adapter abstraction plan tied to Task 62. Create or update an adapter contract document, for example `docs/aegis/agent-adapter-contract.md`, and update `templates/registry/agent-compatibility-matrix.json` only where the schema already supports it. Document Claude as the default implemented adapter and Codex/Gemini/other agents as future/planned adapters. For each adapter, record entry files, hook capability, MCP registration path, mutation surfaces, managed/owned paths, protected paths, enforcement level (`mechanical`, `gated`, `policy-only`, or `planned`), evidence model, limitations, and required verification commands. This is a plan and contract task for non-Claude adapters, not a full Codex/Gemini runtime implementation. 9. Add TestPyPI/PyPI readiness only as documentation input. Update `docs/aegis/release-policy.md`, `docs/aegis/distribution.md`, and packaged mirrors with a checklist for metadata, versioning, artifact inspection, provenance, signing/checksums, dry-run validation, and publication decision records. Do not run `twine upload`, do not publish to TestPyPI/PyPI, and do not mark publication ready unless the live matrix and local artifact proofs are satisfied. 10. Maintain source/package parity. Any change to installer core, rendered guidance, docs, hook scripts, or workflow templates must be mirrored into packaged assets and validated by existing package-data/release tests. Update `V1_TOOL_NAMES`, `PROMPT_NAMES`, docs, and tests together so MCP discovery remains deterministic. 11. Produce a short roadmap artifact, preferably under the Task 122 active work-tracking `designs/` folder and referenced from `docs/aegis/live-acceptance-matrix.md`, with three sections: Task 121 immediate fixes already handled or assumed; Task 122 workflow guidance/portability deliverables; deferred release publication and future adapter implementation work. Acceptance requires that roadmap to explicitly separate Task 121 repair work from longer-term release and adapter work.
