# Task ID: 117

**Title:** Add Aegis Closeout Gate and Live-Agent Completion Flow

**Status:** pending

**Dependencies:** 116 ✓

**Priority:** medium

**Description:** Implement a project-local Aegis closeout command plus Claude runtime instructions and gates so an agent cannot treat portable Aegis work as complete until readiness, tracking, plan, verification, evidence, handoff, and optional Git guidance checks all pass mechanically.

**Details:**

Codebase analysis summary: Aegis is a Python 3.11 package defined in pyproject.toml with the console entry point aegis = aegis_foundation.cli:main and repository wrapper support in scripts/codex-task. Core installer/runtime behavior lives in scripts/_aegis_installer.py and is packaged under aegis_foundation/assets/scripts/_aegis_installer.py. Existing commands include inspect, plan-install, status, install, verify --strict, certify-release, kickoff, and log in aegis_foundation/cli.py, with matching scripts/codex-task aegis handlers. Kickoff/log state is stored under .aegis/state/current-work.json and .aegis/state/pending-tracking.json, workflow files are scaffolded from aegis_foundation/assets/templates/aegis/workflow/{session,plan,tracker,findings,decisions,implementation,changelog,handoff}.md, and Claude hook behavior is enforced through .claude/scripts/gate_lib.py plus packaged copies under aegis_foundation/assets/.claude/scripts/. Current strict verification in scripts/_aegis_installer.py validates manifest, local shim, workflow templates, current work, branch alignment, tracking surfaces, pending tracking, Claude hook registration, protected path guards, and optional Taskmaster/Serena integration state. Existing regression coverage is concentrated in tests/meta_workflow_guard/test_aegis_installer.py, tests/meta_workflow_guard/test_aegis_mcp_e2e_targets.py, and tests/claude_adapter/test_*.

Implement the closeout flow in the shared Aegis core, not separately per adapter. Add a closeout function in scripts/_aegis_installer.py, for example closeout(target_dir, *, source_root, require_clean_git=False, commit_guidance=True), and expose it through aegis_foundation/cli.py as aegis closeout --target-dir . plus scripts/codex-task aegis closeout. If MCP already exposes strict verify as an optional flag, consider adding aegis.closeout only if it fits the existing aegis_mcp.server V1 tool schema and acknowledgement conventions; otherwise keep MCP follow-up explicit and document the CLI as the portable authority.

The closeout command should be read-mostly but may write a closeout report and, if all gates pass, update Aegis-owned workflow state. It must run or directly invoke the same readiness logic used by .claude/scripts/readiness.sh and require READY. It must require .aegis/state/current-work.json to exist and represent active in-progress work. It must fail if .aegis/state/pending-tracking.json contains events or invalid JSON. It must enforce required plan steps in order: plan-step-scope completed before implementation, plan-step-implement completed after a task-scoped mutation evidence entry, and plan-step-verify completed only after strict verification evidence is logged. It must run strict verification through the existing verify(..., strict=True) core path or the installed local shim and require status passed. It must require current plan and tracker agreement for required steps rather than trusting only one file.

Add evidence cross-reference checks. For each required evidence item, especially task-scoped source changes and verification commands/reports, verify that the evidence token appears in sessions/current, the active TRACKER.md, IMPLEMENTATION.md, CHANGELOG.md, HANDOFF.md, and plans/current. Reuse existing S:W:H:E token conventions from log_work: [S:<date>|W:task<id>-<slug>|H:<handler>|E:<evidence>]. Closeout should detect missing surfaces with precise check IDs such as closeout.evidence.session, closeout.evidence.tracker, closeout.evidence.implementation, closeout.evidence.changelog, closeout.evidence.handoff, and closeout.evidence.plan. It should not count Serena memory or Taskmaster entries as required evidence unless current-work explicitly marks those integrations required; Taskmaster and Serena remain optional.

Improve handoff semantics. The current HANDOFF.md template has Current State, What Was Done, Current Issues/Blockers, Next Steps, Important Context, and Progress Log sections, while aegis log currently appends progress entries. Add closeout validation that HANDOFF.md contains semantic current state and next steps that reflect the completed task rather than only appended logs. Implement a conservative parser/checker that requires non-placeholder content under Current State and Next Steps, requires the changed source evidence and strict verification evidence to be mentioned outside Progress Log, and rejects kickoff-only default text such as 'Task <id> has been kicked off through Aegis' as the final current state. Consider adding an aegis handoff-update or closeout --update-handoff option only if it can update Aegis-owned handoff sections deterministically; otherwise fail with actionable remediation telling the agent to use aegis log plus a handoff update before closeout.

Update Claude instructions and hooks so completion cannot remain advisory. Extend the installed Claude runtime entrypoint rendered by _render_claude_entrypoint in scripts/_aegis_installer.py, the packaged assets, and docs/aegis/invocation-contract.md to define the normal loop as readiness -> scope log -> scoped mutation -> pending tracking -> implementation log -> verification log -> strict verify -> closeout. Add .claude/commands/closeout.md and package it into installed targets if command files are managed for Claude. Update .claude/settings.json allowlist rendering to permit aegis closeout and ./.aegis/bin/aegis closeout. If a Stop hook can mechanically nudge or block final-task-complete phrasing, add a narrow closeout gate script or extend gate_lib.py with a Stop-time check that blocks when the agent attempts to end while plan-step-verify is complete but closeout has not passed; avoid blocking ordinary pauses that still have legitimate next steps. Any hook changes must be mirrored in root .claude/scripts and aegis_foundation/assets/.claude/scripts, and strict verification must validate the new hook/command registration where appropriate.

Produce a machine-readable report at .aegis/reports/closeout-report.json with schema_version, status, closed_at or checked_at, target_root, current_work task, readiness result, strict_verify report path/status, plan step statuses, evidence matrix by surface, handoff semantic checks, pending tracking summary, optional integration summary, git guidance, and failures. On success, consider updating current-work status to closed or closeout-passed while preserving enough state for strict verify and future inspection; if changing status from in-progress would break readiness or strict verify, record closeout_passed_at and closeout_report instead of changing the active status until a separate archive task exists.

Add optional Git/PR guidance without defaulting to gac. Align with templates/handlers/operators/git/create-commit-message.md and templates/shared/tools/tool-selection-matrix.md: recommend normal git add, git commit, git push, and gh pr create when the user delegates Git work and auth is available; mention gac only as an explicit user-requested legacy/manual convenience. Closeout should print or report suggested commands, not run commits or PR creation by default. Guard/docs should reject any new Aegis closeout guidance that makes gac the default.

Keep implementation portable. The closeout command must work in installed target projects that have no .taskmaster and no .serena directories. It may report those integrations as detected/absent and required/optional based on .aegis/state/current-work.json, following the same semantics already used by readiness and strict verification. Do not add mandatory Taskmaster set-status or Serena memory writes to the portable Aegis closeout path.
