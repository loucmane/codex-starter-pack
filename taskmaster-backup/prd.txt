# Overview
Codex is inheriting the CLAUDE template operating system. The migration project must eliminate the legacy monolithic template files, finish the modular breakdown, and institutionalize the enforcement + evidence tooling so every agent (Codex, Claude, Taskmaster, Serena) shares a single source of truth. The work began in the FPL MCP repository (Python template compatibility analysis, SSOT scanner suite, enforcement hooks) and is partially migrated into this Codex repo. Remaining effort spans reference remediation, modular template authoring, guard automation, CI integration, documentation, workflow protocols (including compaction), cross-agent compatibility, security, cost governance, and long-term maintenance.

# Core Features
1. **Single-Source-of-Truth Template Library**
   - Replace monoliths (WORKFLOWS.md, PATTERNS.md, etc.) with modular directories.
   - Maintain authoritative index files while enforcing canonical locations.
   - Provide migration-aware metadata (fully migrated, partial, not migrated) for every file.
2. **SSOT Scanner + Report Suite**
   - Python tooling under `scripts/template-ssot-scanner/` scans, analyzes, and fixes references.
   - Outputs JSON + scripts (`apply_reference_fixes.py`, `safe_reorganize.py`, etc.) with versioned metadata.
   - Supports include/exclude filters, heading validation, duplicate detection, migration detector, configurable rule severities.
3. **Enforcement Helpers (`codex-task`, `codex-guard`)**
   - CLI utilities to scaffold S:W:H:E logs and validate compliance before handoff/commit (with planned auto-fix mode and guard allowlists).
   - Integration planned for git hooks, pre-commit, CI, and operational runbooks.
4. **Workflow + Session Management System**
   - Standardized work-tracking (7 canonical files), compaction protocol, Serena memories, session logs, Taskmaster tasks.
   - Migration state persists across compactions, CI runs, and repository sync via handoffs, Serena memories, and Taskmaster snapshots.
5. **Automation & Integration Layer**
   - CI jobs to run scanners, guard, prd-lint, performance tests, and Taskmaster validation.
   - Hooks for Taskmaster, Serena, Sequential Thinking, Codex wrapper, and guard enforcement.
6. **Template Authoring Standards**
   - Handler frontmatter schema enforced for every template (id, type, category, dependencies, triggers, exports, version, status).
   - S:W:H:E guidance, discovery fallback chain (modular → compatibility map → legacy → Serena → error), and lifecycle management (creation → deprecation → archival).

# Risks and Mitigations
- **Scanner Breakdown**: If scanner mis-detects references, sample manual validation pre-phase; maintain allowlist.
- **Agent Incompatibility**: Agents unable to load modular templates; maintain monolith escape hatch until proven.
- **Performance Cliff**: Modular templates slow down load times; enforce performance gate; implement caching.
- **Context Explosion**: Modular templates increase token usage; enforce token usage gate, prune content.
- **Template Discovery Failure**: Dual-path fallback chain ensures discovery; run exhaustive tests.
- **Budget Overrun**: Monitor API usage; implement caching; set alerts at 80% budget.
- **Security Vulnerabilities**: Path traversal or injection; sanitize inputs; security audit before gates.
- **Cross-Repo Drift**: Daily sync checks, auto PRs, conflict resolution playbook.
- **CI/CD Instability**: Guard/scanner auto-fix errors; maintain bypass protocol with approval.
- **Template Lifecycle Neglect**: Implement lifecycle policies; review usage monthly.

# Template Lifecycle Management
## Creation
- PR review required; guard validation must pass; documentation updated.
## Naming Conventions
- Handlers: `category-action-context.md` (e.g., `session-start-new.md`).
- Operators: `op-verb-target.md` (e.g., `op-validate-references.md`).
- Patterns: `pattern-workflow-variant.md`.
- Lowercase with hyphens; no spaces; version captured in frontmatter, not filename.
- Test/support files use `.test.md` suffix and are allowlisted in guard config.
## Deprecation Process
1. Mark as deprecated in frontmatter.
2. Add migration notice pointing to replacement.
3. Grace period 30 days (compatibility map redirects).
4. Move to `templates/deprecated/`.
5. Remove after 90 days.
## Versioning & Conflict Resolution
- Templates versioned via frontmatter `version` field and git history; major changes tagged in git.
- If agents require different versions: compatibility layer serves both; Taskmaster task opened; deprecation warning logged; forced upgrade after grace period.
- Change advisory board reviews breaking changes before merge.

# Error Recovery Patterns
| Error | Detection | Recovery | Fallback |
|-------|-----------|----------|----------|
| Template not found | Guard/scanner 404 | Check compatibility map | Use legacy monolith | Manual update
| Scanner timeout | >30 s execution | Use cached results, rerun | Manual review | Increase resources
| Guard false positive | Known pattern flagged | Apply allowlist rule | Override with justification | Fix guard rule
| Compaction loss | Missing memory | Restore from sessions/Serena | Reconstruct via git | Document gap
| API rate limit | HTTP 429 | Exponential backoff | Switch to cached/offline mode | Alert, pause automation
