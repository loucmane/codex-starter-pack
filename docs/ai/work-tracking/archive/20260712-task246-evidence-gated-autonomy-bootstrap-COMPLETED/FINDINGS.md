# Findings

- 2026-07-12 — PR #261 passed every pull-request check because `GITHUB_HEAD_REF` preserved `feat/task-245-*`, but both protected-main guards failed at `plan sync` because Task 244 intentionally rejects branches without a task ID.
- 2026-07-12 — The same early source-state failure caused `test_guard_flags_placeholder_handler` to fail on Python 3.11 and 3.12: guard stderr reported no ACTIVE tracker before placeholder validation could run.
- 2026-07-12 — The initial assumption that Auto-review fully handled Codex escalation was falsified during publication: `workspace-write` protects `.git` as read-only, and an exhausted Auto-review quota turned routine staging into another owner boundary. Repository delivery authority and local execution authority both need persistent, separately tested contracts.
- 2026-07-12 — Blog's dogfooded workflow proves the core privileged pattern: evaluate exact-head metadata and complete file inventories with policy code checked out from protected `main`, never execute candidate PR code, and reserve policy/authority paths for attended review.
- 2026-07-12 — GitHub's workflow-run REST payload can report an empty `pull_requests` array even for a known pull-request run; resolving candidates through `commits/{head_sha}/pulls` and requiring exactly one open exact-head match avoids relying on that lossy association.
- 2026-07-12 — Pull-request workflow runs expose the branch head SHA through the Actions API, so required workflow evidence can be joined deterministically by workflow name, `event=pull_request`, and exact head.
- 2026-07-12 — A workflow-token merge needs an explicit `repository_dispatch` lane for reliable post-merge validation; the dispatch carries the returned merge SHA and guard workflows check out that exact object.
- 2026-07-12 — The source `CODEX.md` is itself an installer input. Updating generated continuation wording without updating its managed block breaks fresh-install idempotence on Codex-enabled targets.
- 2026-07-12 — Merge autonomy alone is insufficient for unattended operation: hardcoded repair, closeout, task-transition, push/PR, and CI-remediation confirmation text recreates the same chat boundary before delivery. Those capabilities must be explicit policy fields and fail closed independently.
- 2026-07-12 — Exact-SHA checkout for `repository_dispatch` is detached HEAD by default. Post-merge source validation needs both the exact merge object and local default-branch identity, so the workflows validate the payload then run `git switch -C <default> <merge-sha>` before guards.
- 2026-07-12 — Codex 0.144 permission profiles can reopen `.git` within the primary workspace while inheriting `:workspace` protection for `.codex` and `.agents`; a finite network allowlist plus `approval_policy = "never"` removes recurring prompts without using `danger-full-access`. Linked-worktree common Git directories remain outside that relative grant.
- 2026-07-12 — Hosted witness CI does not consume the plan's explicit scope list when untracked scope records are unavailable; it uses tracked `.aegis/brief.json` fallback globs. New root authority files and `.codex` configuration therefore require explicit brief scope entries or they fail diff accounting.
- 2026-07-12 — Making `.git` writable removes routine approval friction but also removes one local sandbox barrier. A mode-independent PreToolUse deny materially reduces accidental destructive Git/GitHub operations, but command parsing cannot substitute for OS mediation: arbitrary interpreters and direct `.git` writes remain bounded by the sandbox, protected branch, attended authority paths, and trusted-base delivery policy.
